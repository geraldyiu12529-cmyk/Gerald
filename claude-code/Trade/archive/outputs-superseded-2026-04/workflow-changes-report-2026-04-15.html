Workflow Changes Report

2026-04-15  |  What changed, why, and what it means for your trading
Bottom line: Your workspace was well-designed but operationally gated — the methodology correctly blocked trades because data couldn't flow into the scorecard. These six changes fix the plumbing so the system can actually fire when signals align, while also cutting waste (no more 8 versions per day) and keeping your files manageable over time.

1. Audit-Addition Data Pipeline

PROBLEM Your book was frozen

On 2026-04-14, you added three Grade A variables to the methodology (residual momentum, intermediary capital, basis-momentum). These are powerful signals — residual momentum alone would unblock 12 single-stock T-scores, and intermediary capital would resolve the EWY fail-loud caveat. But none of the three had an automated way to get data. The trade-rec tried web searches during its run, couldn't find usable values, and correctly marked them MISSING. The fail-loud rule then blocked every candidate: EWY at +3 was treated as a near-miss, TSM/AMZN/META couldn't be scored at all. The methodology was working as designed — but with no data flowing in, you had a perfectly disciplined system that could never trade.

SOLUTION Dedicated compute task

Created audit-data-compute-750pm — a scheduled task that fires at 19:50 UTC+8 every weekday, 10 minutes before the daily brief. It uses web search to fetch source data (Fama-French 5-factor returns from Kenneth French's library, NY Fed primary dealer statistics, commodity front-curve settlement data), then runs a Python script (scripts/compute_audit_additions.py) that does the actual math: 12-month rolling OLS regressions for residual momentum, z-score computation for intermediary capital, and 4-week/12-week slope changes for basis-momentum. Results are written to audit-data-staging-YYYY-MM-DD.md — a staging file that the brief and trade-rec read at startup.

IMPACT ON YOU

This is the single most consequential change. Once the compute task starts producing non-MISSING values, the entire single-stock book becomes scorable (12 names get T-scores), the EWY intermediary-capital caveat can be resolved, and the commodity S divergence-cap can actually fire. Near-misses like TSM, AMZN, and META at S+1/C+1 become promotable to |Sum|=+3 if residual momentum reads positive. The system goes from "correctly frozen" to "fully operational."

2. Delta-Check Gate on Trade Rec

PROBLEM Eight identical versions on Day 1

On 2026-04-14, the trade rec reached v8 — each version running the full 8-step methodology, producing a new HTML report, appending several lines to Memory.md §8, and reaching the same conclusion: "0 actionable signals, no trade." Between v5 and v8, no upstream data changed. Each rerun was pure waste: more compute, more Memory.md bloat, more version noise in the audit trail. When you look back in a month, v6/v7/v8 contain zero incremental information.

SOLUTION Timestamp comparison before rerun

Updated the trade-rec task prompt with a delta-check gate. Before running the full 8-step, it now compares the modification timestamps of all five upstream artifacts (brief, news, US close snapshot, weekly review, audit-data staging) against the existing trade-rec file. If nothing is newer, it logs a one-line "no-change revalidation" note and exits. The full 8-step only runs when there's actually new data to score.

IMPACT ON YOU

You'll see one or two trade-rec versions per day instead of eight. Memory.md §8 stays concise. The audit trail is cleaner — every version represents a genuine data update, not a redundant rerun. And each evening's pipeline finishes faster because the rec doesn't re-score unchanged data.

3. Weekly Memory Condensation

PROBLEM Memory.md was growing unsustainably

Memory.md §8 (Lessons & Corrections) hit 180+ lines after one day. Every trade-rec version, every brief refresh, every audit-review fire added a multi-line entry. At that rate, §8 would exceed 1,000 lines within a week. Since every session startup must read Memory.md in full, this creates a concrete cost: longer load times, more context consumed on the startup reads, higher chance of hitting context limits on complex analysis sessions.

SOLUTION Sunday condensation in the weekly review

Created weekly-regime-review-sunday-6pm-v2 (replacing v1, which is now disabled). After writing the weekly review, it runs a new condensation step: takes all of the prior week's individual §8 entries, rolls them into a 3-5 sentence summary paragraph, and archives the verbose originals to archive/memory-log-YYYY-WNN.md. The full detail is preserved in the archive — it just moves out of the hot startup path.

IMPACT ON YOU

Memory.md stays lean week over week. Session startups read a concise summary for each past week instead of dozens of individual log entries. If you ever need the verbose detail (for a methodology review, an audit, or debugging), it's in the archive with the ISO week number. The 2026-10-14 six-month review will still have full evidence — just filed efficiently.

4. Pipeline Scheduling Overhaul

PROBLEM Collision risk and weekend waste

The brief and news tasks both fired at the exact same cron minute (20:00). In practice, jitter separated them, but this was fragile — if both hit web search simultaneously, they'd compete for rate limits. Additionally, all five daily tasks fired every day including Saturday and Sunday, when US markets are closed and most data sources return stale readings. This produced empty or recycled briefs on weekends that added noise without value.

SOLUTION Staggered times, weekday-only, plus preflight check

Restructured the entire pipeline with proper sequencing and a new preflight validation step:

19:50 ──▶ audit-data-compute (NEW)   fetches & computes audit-addition variables
20:00 ──▶ market brief   reads audit-data staging + all Grade A pulls
20:10 ──▶ news capture   geopolitics, macro, corporate, crypto, flash events
20:15 ──▶ preflight check (NEW)   validates all upstream artifacts exist
20:25 ──▶ trade rec   delta-check → full 8-step if new data

All Mon-Fri only. Weekend tasks: weekly review (Sun 18:00), workspace tidy (Sun 21:00).

IMPACT ON YOU

No more wasted runs on weekends. Each task fires in a clean sequence with no collision risk. The new preflight check at 20:15 catches missing upstream files before the trade rec runs — so if the brief failed silently, you get a clear diagnostic instead of a trade rec with unexplained data gaps. The audit-data compute at 19:50 means the brief at 20:00 can read pre-computed audit-addition values instead of trying (and failing) to web-search them mid-run.

5. Unified Retention Logic

PROBLEM Two systems, one set of files

The master Retention Policy.md defined a 3-tier archival system (live → archive → digest), and the news-events/README.md defined its own independent 7-day/30-day rules with a separate archive-summary.md mechanism. These overlapped on the same news files. The Sunday workspace-tidy task follows the master policy; if someone later built a news-specific cleanup task following the README rules, news files could be moved, condensed, or deleted inconsistently.

SOLUTION Single owner

Rewrote the retention section in news-events/README.md to explicitly defer to the master Retention Policy.md. Added a corresponding news-file clause to the master policy so there's no ambiguity. One system, one cleanup task, one set of rules.

IMPACT ON YOU

Minimal day-to-day impact — this is a preventive fix. It ensures that as your workspace ages and the tidy task starts actually archiving files (from week 2 onward), news files won't be caught between conflicting rules. You won't have to debug a situation where a news file was deleted by one system but expected by another.

6. CLAUDE.md Deduplication

PROBLEM Same information in three places

The audit-addition rules were written in full in three locations: CLAUDE.md (8 lines), Methodology Prompt.md (the authoritative source, with reconciliation notes and Top-25 entries 26-28), and Data Sources.md (pull paths). The CLAUDE.md copy duplicated the mechanism description, the divergence-cap rule, the double-counting gate, and the fail-loud rule — all of which are already definitive in the Methodology Prompt. Triple-duplication means triple the maintenance cost when a rule changes, and triple the risk of drift between copies.

SOLUTION Pointer instead of copy

Reduced the CLAUDE.md audit-addition section from 8 lines of duplicated rules to a 3-line pointer: names the three variables, points to the Methodology Prompt entries and Data Sources for the full rules, and references the new audit-data-staging file. Also added the staging file and the compute script to the CLAUDE.md file index so future sessions know they exist.

IMPACT ON YOU

Session startup reads are slightly faster (less text to process). More importantly, when you eventually refine a rule — say, adjusting the divergence-cap threshold or the double-counting gate — you only need to edit it in one place (the Methodology Prompt), not three. Single source of truth.
